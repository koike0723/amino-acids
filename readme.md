# 【チーム開発課題】キャリアコンサルティング予約システム 要件定義書（RFP）

## 1. システムの目的・概要
本システムは、リカレントスクール福岡校におけるキャリアコンサルティングのスケジュール管理および受講生の予約手続きをデジタル化・効率化するWebアプリケーションである。
現状、紙媒体や手作業で行われているスケジュール調整をシステム化することで、事務局の管理コスト削減と、受講生の利便性向上を目指す。

## 2. 現状の課題（As-Is）とシステム化のねらい（To-Be）
現在、キャリコンのスケジュール策定業務において以下の課題が発生しており、本システムでの解決・効率化を目指す。

*   **アナログな情報伝達と手作業の転記:** 予約や変更・キャンセル申請がすべて「紙媒体」で行われており、それを事務局側で「手作業でExcelのスケジュール表に転記・作成」しているため、多大な工数とミスのリスクがある。
*   **キャンセルの「穴埋め調整」の負荷:** 変更・キャンセルや退校などで当初のスケジュールに穴が空いた場合、キャリアコンサルタントの「待機時間」が発生してしまう。これを防ぐため、例えば15:00枠が空いた際に「別のラインの同15:00枠から生徒を移動させる」といったパズルのような調整を手動で行っている。
*   **マスタ未連携によるデータ不整合:** 生徒マスタとスケジュール表が紐づいておらず独立しているため、退校・就職した生徒の情報がいつまでもスケジュール上に残ってしまうケースがある。

## 3. 対象となる業務とルール
リカレントスクール福岡校では、以下の2種類のキャリアコンサルティング（面談）が実施されており、それぞれ運用ルールが異なる。

### 3.1. 実施体制（複数ライン稼働）
*   **ライン（担当者・教室）の並行稼働:** 同一日に複数のキャリアコンサルタントがそれぞれの教室（ライン）で同時に面談を実施する。
    *   （例：コンサルタントAは6A教室で10:00〜16:00、コンサルタントBは6B教室で10:00〜15:00、コンサルタントCは6C教室で11:00〜16:00など、ラインごとに稼働時間帯が異なる場合がある）
*   システム上、同じ時間帯（例：10:00枠）に複数の予約可能枠（ライン数分）が存在することを考慮した設計が必要となる。

### 3.2. キャリコン（必須面談）
*   **対象:** 「求職者訓練」カテゴリのコースを受講する訓練生全員（受講必須）。
*   **実施枠:** 6ヶ月の訓練期間中に数回（土曜日の固定枠）実施される。回数はコース・開講時期によって変動する。
*   **固定枠:** 10:00 / 11:00 / 12:00 / 14:00 / 15:00 / 16:00 のいずれか（各60分）。
*   **スケジューリング:** 訓練開始前（受講生確定時）に、**学校スタッフが全6回分のスケジュールを通しで作成し、各受講生に割り当てる。**

### 3.3. キャリコンプラス（任意面談）
*   **対象:** 全ての訓練コースの受講生（任意参加）。通常のキャリコンと併用して受講可能。
*   **実施枠:** 学校側が任意に指定した土曜日の固定枠（時間は必須キャリコンと同じく10:00〜16:00の60分枠の予定）。
*   **スケジューリング:** **受講生自身がシステム上で予約を入れる。**
*   **予約ルール:**
    *   実施日の1ヶ月前から予約可能。
    *   1人につき**「月に1回まで」**の予約制限がある。（例：同日に10:00枠と14:00枠を予約することは不可）。

## 4. アクター（システム利用者）
本システムを利用する権限（ロール）は以下の2種類とする。

| アクター名 | 役割・概要 |
| :--- | :--- |
| **訓練生** | 自身のスケジュールの確認、およびキャリコンプラスの予約・キャンセル・必須キャリコンの変更申請を行う。 |
| **学校スタッフ** | システム全体の管理、コース・アカウントの設定、キャリコン（必須）のスケジュール割り当てと管理を行う。 |

## 5. 機能要件（ユースケース一覧）

### 5.1. 訓練生（受講生）向け機能
1.  **[必須キャリコン] スケジュール確認:**
    *   事務局から割り当てられた自身の「キャリコン（必須）」の実施日時一覧を確認できること。
2.  **[必須キャリコン] スケジュール変更・キャンセル申請:**
    *   割り当てられた日時に都合がつかない場合、システム上から事務局へ日時変更やキャンセルの申請（変更届の提出）ができること。
    *   ※システム上で即座に変更が確定するのではなく、事務局への「申請」という扱いになる。
3.  **[任意キャリコンプラス] 空き枠検索:**
    *   実施日の1ヶ月前から、予約可能な「キャリコンプラス」の空き枠を確認・検索できること。
4.  **[任意キャリコンプラス] 面談予約:**
    *   空き枠を選択し、予約を確定できること。（※「1人月1回まで」の制限を満たしているかチェックが必要）
5.  **[任意キャリコンプラス] 予約キャンセル:**
    *   自身が予約した「キャリコンプラス」の枠を取り消すことができること。

### 5.2. 学校スタッフ（管理者）向け機能
1.  **マスタ管理:**
    *   訓練生のアカウント（ログインID/パスワード等）およびステータス（在校・退校・就職等）を作成・変更・管理できること。
    *   訓練コース（「求職者訓練」に該当するかどうかの情報を含む）を作成・管理できること。
    *   ※生徒のステータスが「退校・就職」に変わった際、システム上のスケジュール割り当て情報と連動し、不要な予約が自動的に取り消される（またはアラートが出る）仕組みが存在すること。
2.  **[必須キャリコン] スケジュールの一括作成・割り当て:**
    *   訓練開始前に、対象コース（求職者訓練）の受講生に対して、スケジュール（月1回・土曜日・指定の固定枠）を一括でシステム上に作成・割り当てできること。
3.  **[必須キャリコン] 変更・キャンセル申請の処理:**
    *   訓練生から提出された「変更・キャンセル申請」を確認し、承認（スケジュールの再割り当てや取り消しの反映）などの処理ができること。
4.  **[必須キャリコン] 個別スケジュール調整および穴埋め処理:**
    *   申請やその他の理由に基づき、特定の訓練生の面談日時を個別に変更できること。
    *   キャンセルや退校等で発生した「スケジュールの空き枠（穴）」に対し、キャリアコンサルタントの待機時間を減らすため、別の訓練生を移動させるなどの調整が容易に行えること。
5.  **[任意キャリコンプラス] 実施枠の公開:**
    *   面談を実施する土曜日の日程を指定し、予約可能な固定枠（10:00/11:00/12:00/14:00/15:00/16:00等）をシステム上に公開できること。
6.  **[任意キャリコンプラス] 予約状況照会:**
    *   公開した枠に対して、どの訓練生が予約を入れているか（あるいは空き・キャンセルになっているか）を一覧で確認できること。

## 6. チーム開発における課題（技術的な挑戦ポイント）
本ドキュメント（要件定義書）に基づき、各開発チームは以下の設計・実装工程を行うこと。
特に**「現状の課題（As-Is）を、システムを使ってどうスマートに解決するか」**がソフトウェアの価値・評価ポイントとなる。

1.  **要件分析・UI/UX設計（画面遷移図と画面設計）**
    *   紙で行っていた申請を、訓練生がスマホやPCからどう迷わず直感的に操作できるか？
    *   「スケジュールの穴埋め調整」のアナログなパズル作業を、事務局がどうやればミスなく簡単に行えるか？（例：ドラッグ＆ドロップ操作、空き枠や待機アラートの可視化など）
    *   **複数ライン（並行する別々の教室・担当者）のスケジュールを、どうすれば管理者が見やすく、かつ直感的に管理できる画面になるか？**
2.  **データモデリング（DB/ER図設計）**
    *   アカウント、コース、固定枠情報に加え、**複数ライン（面談枠の入れ物）**、および「必須のスケジュール」と「任意の予約」をデータベース上でどのように整理・連携するか？
    *   「生徒マスタ（ステータス情報）」と「スケジュール」をどう連動させ、データの不整合（退校・就職者の予約残存など）を防ぐか？
3.  **システム開発（プログラミングとテスト）**
    *   設計に基づき、Webアプリケーションとして要件を満たすシステムを実装すること。
